# 开发流程

本项目采用轻量版本控制流程：每个阶段保留清晰提交，PR 不再是每个任务的强制要求。

Codex 和 Claude 的任务协作入口在 `Agent.md`。活跃任务、交接记录和任务模板都以该文件为准。

本机开发默认不使用 Docker。Claude 在本机验证时应直接运行后端、前端和测试命令；Docker 只作为最终服务器部署打包方式，在有 Docker 的机器上验证。

## Claude 执行规则

Claude 可以在本地仓库中直接工作，但每个任务完成后必须留下干净、可审查、可回退的提交。

每个任务按下面流程执行：

1. 从最新稳定状态开始。
2. 每次只处理一个边界清晰的任务。
3. 完成后运行相关测试和检查。
4. 用清楚的提交信息提交本次任务。
5. 总结改了什么、怎么验证、还有什么风险。

任务较大、风险较高、需要试验，或可能需要整体丢弃时，再创建独立分支。

只有下面情况才需要 PR：

- 你明确要求创建 PR。
- 改动较大，需要单独审核。
- 涉及数据库迁移。
- 涉及登录、权限、token 加密或部署。
- 需要把整块改动作为独立单元审核和回退。

## 分支命名

需要分支时，分支名要简短、清楚，例如：

- `feat/backend-scaffold`
- `feat/api-keys`
- `feat/mail-fetch-graph`
- `fix/oauth-refresh-errors`
- `test/mail-account-claim`

## PR 要求

使用 PR 时，需要说明：

- 改了什么。
- 为什么要这样改。
- 做了哪些测试。
- 是否涉及数据库迁移或部署变化。
- 如果是前端改动，必要时附截图。

## 回退策略

每个完成的任务都应该有自己的提交，或一小组相关提交。如果某次改动出问题，可以直接 revert 对应提交。

对于较大或风险较高的改动，优先使用功能分支或 PR，这样整块改动可以作为一个单元审核和回退。

数据库迁移需要特别谨慎。能兼容旧版本时，优先做向后兼容迁移；不能兼容时，PR 里必须写清楚回滚方式。

## main 分支

`main` 分支代表当前已知的稳定状态。小任务可以通过清晰提交直接落到 `main`；较大或高风险任务先用分支。
